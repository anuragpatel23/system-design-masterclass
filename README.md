## рҹ“– What is System Design Masterclass?

Most system design resources give you theory. System Design Masterclass gives you **implementation** alongside it.

Every topic in this vault follows the same pattern:
1. **Concept** вҖ” the "why" explained clearly
2. **Design** вҖ” diagrams, trade-offs, and decision points
3. **Implementation** вҖ” working code or pseudocode
4. **Real-world reference** вҖ” how companies like Netflix, Uber, Discord solved it

No fluff. No shallow overviews. Just deep, honest engineering.

## рҹ”‘ Key Concepts Quick Reference

<details>
<summary><strong>рҹ“Ҡ CAP Theorem</strong></summary>

A distributed system can guarantee only **2 of 3**:

- **C**onsistency вҖ” every read gets the most recent write
- **A**vailability вҖ” every request gets a response
- **P**artition Tolerance вҖ” system works despite network failures

Since partition tolerance is non-negotiable in distributed systems, the real choice is **CP vs AP**.

вҶ’ [Full deep dive](01-foundations/cap-theorem/README.md)

</details>

<details>
<summary><strong>вҡЎ Caching Strategies</strong></summary>

| Strategy | When to use | Risk |
|----------|-------------|------|
| Cache-aside | Read-heavy, occasional updates | Cache miss on cold start |
| Write-through | Consistency critical | Write latency |
| Write-behind | Write-heavy workloads | Data loss on crash |
| Read-through | Simplicity needed | Cache miss penalty |

вҶ’ [Full deep dive](02-building-blocks/caching/README.md)

</details>

<details>
<summary><strong>рҹ—„пёҸ SQL vs NoSQL вҖ” When to choose what</strong></summary>

**Choose SQL when:** ACID compliance needed В· Complex queries В· Structured, relational data

**Choose NoSQL when:** Massive scale В· Flexible schema В· High write throughput В· Key-value / document / graph data models

вҶ’ [Full deep dive](02-building-blocks/databases/sql-vs-nosql/README.md)

</details>

<details>
<summary><strong>рҹ“¬ Message Queue Patterns</strong></summary>

- **Point-to-point** (Queues) вҖ” one consumer per message (SQS)
- **Pub/Sub** вҖ” multiple consumers per message (Kafka, SNS)
- **Dead Letter Queue** вҖ” handle failed messages gracefully

вҶ’ [Full deep dive](02-building-blocks/message-queues/README.md)

</details>

## рҹҺҜ Interview Framework (RESHADED)

When answering any system design question, use this checklist:

```
R вҖ” Requirements        Clarify functional & non-functional needs
E вҖ” Estimation          Back-of-envelope: users, QPS, storage
S вҖ” Storage             Choose the right database(s) and why
H вҖ” High-Level Design   Draw the big picture first
A вҖ” APIs                Define key endpoints
D вҖ” Data Model          Schema, entities, relationships
E вҖ” Extended Design     Deep dive: caching, sharding, failover
D вҖ” Discussion          Trade-offs, bottlenecks, future scaling
```

вҶ’ [Full interview playbook](09-interview-prep/interview-framework.md)
